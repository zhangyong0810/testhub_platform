## 第一章 产品概述

### 1.1 产品简介

UI 自动化测试平台是一款基于 Web 的自动化测试管理工具，支持可视化创建和管理测试用例，无需编写代码即可完成自动化测试任务。平台采用 Playwright+Selenium 双引擎驱动，支持多浏览器执行，提供完整的测试报告功能。

### 1.2 核心功能

#### 1.2.1 元素管理
- 支持多种元素定位策略（ID、CSS Selector、XPath、Name、Class 等）
- 元素库集中管理，支持分组和页面关联
- 备用定位器配置，提高元素稳定性
- 元素验证功能，确保定位器有效性

#### 1.2.2 测试用例管理
- 可视化测试用例编辑器
- 拖拽式步骤编排
- 丰富的操作类型（点击、输入、断言、截图等）
- 支持动态变量和数据工厂集成

#### 1.2.3 测试套件管理
- 批量管理测试用例
- 自定义执行顺序
- 多浏览器支持（Chrome、Firefox、Safari、Edge）
- 有头/无头执行模式

#### 1.2.4 定时任务管理
- Cron 表达式定时执行
- 固定间隔执行
- 单次任务调度
- 邮件/Webhook 通知

#### 1.2.5 测试报告
- 自动生成测试报告
- 详细的执行日志
- 截图证据链
- 通过率统计

### 1.3 技术架构

- **前端框架**: Vue 3 + Element Plus
- **后端框架**: Django + Django REST Framework
- **自动化引擎**: Playwright（推荐）、Selenium
- **数据库**: MySQL/PostgreSQL
- **浏览器支持**: Chrome、Firefox、Safari、Edge

## 第二章 快速开始

### 2.1 系统准备

#### 2.1.1 安装依赖

```bash
# 激活虚拟环境
source .venv/bin/activate

# 安装核心依赖
pip install playwright selenium

# 安装 Playwright 浏览器
playwright install chromium firefox webkit
```

#### 2.1.2 下载 WebDriver 驱动

```bash
# 下载 Chrome、Firefox、Edge 的 WebDriver
python manage.py download_webdrivers
```

#### 2.1.3 初始化定位策略

```bash
# 初始化元素定位策略
python manage.py init_locator_strategies
```

### 2.2 创建第一个测试项目

1. 登录系统，进入 **UI 自动化测试** 模块
2. 点击 **项目管理** > **新增项目**
3. 填写项目信息：
   - 项目名称：例如"电商平台测试"
   - 项目描述：描述测试范围和目标
   - 基础 URL：被测系统的基础地址，如 `https://example.com`
   - 项目状态：选择"进行中"
   - 负责人：默认为当前用户
4. 点击 **保存**

### 2.3 快速创建测试用例

#### 步骤 1：创建页面元素

1. 进入 **元素管理** 页面
2. 选择刚创建的项目
3. 点击 **新增元素**
4. 填写元素信息：
   - 元素名称：如"登录按钮"
   - 所属页面：如"登录页"
   - 定位策略：选择"CSS Selector"
   - 定位表达式：如 `.login-btn`
   - 是否唯一：勾选（提高定位准确性）
   - 等待超时：5 秒（默认值）
5. 点击 **保存**

#### 步骤 2：创建测试用例

1. 进入 **用例管理** 页面
2. 选择项目
3. 点击 **新增用例**
4. 填写用例信息：
   - 用例名称：如"用户登录测试"
   - 用例描述：描述测试场景
   - 优先级：选择"高"
5. 点击 **保存**

#### 步骤 3：添加测试步骤

1. 在用例详情页面，点击 **添加步骤**
2. 配置步骤信息：
   - 操作类型：选择"点击"
   - 选择元素：选择之前创建的"登录按钮"
   - 步骤描述：如"点击登录按钮提交表单"
3. 继续添加其他步骤（如填写用户名、密码等）
4. 点击 **保存用例**

#### 步骤 4：执行测试

1. 在用例详情页面，选择执行配置：
   - 测试引擎：Playwright（推荐）
   - 浏览器：Chrome
   - 运行模式：有头模式（可查看执行过程）
2. 点击 **运行**
3. 等待执行完成，查看执行结果和日志

## 第三章 元素管理

### 3.1 元素管理概述

元素是 UI 自动化测试的基础，是页面上可交互的组件。良好的元素管理能提高测试脚本的稳定性和可维护性。

### 3.2 创建元素

#### 3.2.1 基本信息配置

进入 **元素管理** 页面，点击 **新增元素**，填写以下信息：

1. **元素名称**：简洁明了地描述元素功能，如"首页搜索框"
2. **所属页面**：元素所在的页面名称，如"首页"、"登录页"
3. **项目**：选择元素所属的测试项目
4. **元素描述**：详细描述元素的用途和特征

#### 3.2.2 定位策略配置

平台支持多种定位策略：

| 定位策略 | 说明 | 示例 |
|---------|------|------|
| ID | 通过元素 ID 定位（推荐） | `username` |
| CSS Selector | 通过 CSS 选择器定位 | `.btn-primary` |
| XPath | 通过 XPath 表达式定位 | `//button[@type='submit']` |
| Name | 通过 name 属性定位 | `q` |
| Class Name | 通过类名定位 | `container` |
| Tag Name | 通过标签名定位 | `input` |
| Link Text | 通过链接文本定位 | `点击这里` |
| Partial Link Text | 通过部分链接文本定位 | `点击` |

**定位表达式编写技巧**：

1. **CSS Selector 示例**：
   ```css
   /* 类选择器 */
   .btn-primary
   
   /* ID 选择器 */
   #submit-btn
   
   /* 属性选择器 */
   input[type="text"]
   
   /* 后代选择器 */
   .form .input-group input
   
   /* 伪类选择器 */
   button:hover
   ```

2. **XPath 示例**：
   ```xpath
   /* 绝对路径（不推荐） */
   /html/body/div[1]/form/input[2]
   
   /* 相对路径（推荐） */
   //button[@class='submit']
   
   /* 文本匹配 */
   //a[text()='登录']
   
   /* 属性匹配 */
   //input[@name='username']
   
   /* 组合条件 */
   //div[@class='form' and @id='loginForm']
   ```

#### 3.2.3 高级配置

1. **备用定位器**：
   - 当主要定位策略失效时，系统会自动尝试备用定位器
   - 格式：JSON 数组
   ```json
   [
     {"strategy": "xpath", "value": "//button[@id='submit']"},
     {"strategy": "css", "value": "#submit-btn"}
   ]
   ```

2. **是否唯一**：
   - 勾选后，系统会验证定位器在页面中是否唯一
   - 建议对关键元素勾选此选项

3. **等待超时**：
   - 设置元素操作的最大等待时间（秒）
   - 默认值：5 秒
   - 对于加载较慢的元素，可适当增加超时时间

4. **强制操作**：
   - 对 `visibility:hidden` 的元素使用 force 选项
   - 谨慎使用，可能影响测试准确性

### 3.3 元素分组管理

#### 3.3.1 创建元素分组

1. 在元素管理页面，点击 **新增分组**
2. 填写分组信息：
   - 分组名称：如"登录模块元素"
   - 分组描述：描述分组包含的元素范围
   - 父分组：可选择上级分组，形成层次结构
3. 设置排序值，控制分组显示顺序
4. 点击 **保存**

#### 3.3.2 元素归类

1. 编辑元素时，选择所属分组
2. 支持批量修改元素分组
3. 可通过分组筛选查看元素

### 3.4 元素验证

#### 3.4.1 验证定位器

1. 在元素列表中，点击元素右侧的 **验证** 按钮
2. 系统会启动浏览器，尝试定位元素
3. 查看验证结果：
   - ✅ 验证通过：定位器有效
   - ❌ 验证失败：定位器无效，需修改

#### 3.4.2 验证状态说明

| 状态 | 说明 | 处理建议 |
|------|------|---------|
| 有效 | 定位器验证通过 | 可正常使用 |
| 无效 | 定位器验证失败 | 检查定位表达式 |
| 未知 | 尚未验证 | 执行验证操作 |
| 待验证 | 元素已修改，需重新验证 | 执行验证操作 |

### 3.5 元素使用统计

#### 3.5.1 查看使用情况

1. 在元素详情页，查看 **使用次数** 字段
2. 点击 **使用记录** 查看元素在哪些脚本中被使用
3. 统计信息包括：
   - 使用频次
   - 使用位置（脚本名称、步骤号）
   - 使用类型（点击、输入、验证等）

#### 3.5.2 元素优化建议

系统会根据使用情况自动生成优化建议：

- **使用频率高**：建议添加到页面对象中提高复用性
- **从未使用**：考虑是否需要删除
- **定位不稳定**：建议添加备用定位器

### 3.6 元素批量操作

#### 3.6.1 批量导入

支持通过 Excel/CSV 文件批量导入元素：

1. 下载导入模板
2. 填写元素信息（名称、页面、定位策略、定位值等）
3. 上传文件
4. 预览并确认导入

#### 3.6.2 批量导出

1. 筛选需要导出的元素
2. 点击 **导出** 按钮
3. 选择导出格式（Excel/CSV）
4. 下载导出文件

#### 3.6.3 批量删除

1. 勾选需要删除的元素
2. 点击 **批量删除**
3. 确认删除操作

**注意**：正在使用的元素删除需谨慎，可能影响相关测试用例。

## 第四章 测试用例管理

### 4.1 测试用例概述

测试用例是自动化测试的核心，由一系列测试步骤组成。平台提供可视化的用例编辑器，无需编写代码即可完成用例设计。

### 4.2 创建测试用例

#### 4.2.1 基本信息

1. 进入 **用例管理** 页面
2. 点击 **新增用例**
3. 填写用例信息：

| 字段 | 说明 | 示例 |
|------|------|------|
| 用例名称 | 简洁描述测试场景 | 用户登录成功场景 |
| 用例描述 | 详细描述测试目的和前置条件 | 验证已注册用户能成功登录系统 |
| 所属项目 | 选择用例所属项目 | 电商平台测试 |
| 优先级 | 高/中/低，影响执行顺序 | 高 |

4. 点击 **保存** 进入步骤编辑

### 4.3 测试步骤编排

#### 4.3.1 添加步骤

1. 在用例详情页，点击 **添加步骤**
2. 选择操作类型（见下表）
3. 配置步骤参数
4. 填写步骤描述（可选）

#### 4.3.2 支持的操作类型

| 操作类型 | 说明 | 必填参数 | 示例 |
|---------|------|---------|------|
| **点击** | 点击按钮、链接等元素 | 目标元素 | 点击"登录"按钮 |
| **输入文本** | 向输入框填写内容 | 目标元素、输入值 | 在用户名输入框填写"testuser" |
| **获取文本** | 获取元素的文本内容 | 目标元素 | 获取欢迎消息文本 |
| **等待元素** | 等待元素出现 | 目标元素、等待时间 | 等待加载动画消失 |
| **悬停** | 鼠标悬停在元素上 | 目标元素 | 悬停在菜单项上 |
| **滚动** | 滚动到元素位置 | 目标元素 | 滚动到页面底部 |
| **截图** | 截取当前页面 | 无 | 截取登录成功后的页面 |
| **断言** | 验证元素状态或内容 | 目标元素、断言类型、期望值 | 断言登录成功提示可见 |
| **等待** | 固定时长等待 | 等待时间 | 等待 2 秒 |
| **切换标签页** | 切换浏览器标签页 | 标签页索引 | 切换到新打开的标签页 |

#### 4.3.3 步骤参数配置

**1. 点击操作**
```
操作类型：点击
目标元素：登录按钮
步骤描述：点击登录按钮提交表单
```

**2. 输入文本操作**
```
操作类型：输入文本
目标元素：用户名输入框
输入值：testuser
步骤描述：填写用户名
```

**3. 断言操作**
```
操作类型：断言
目标元素：欢迎消息
断言类型：文本包含
期望值：欢迎您
步骤描述：验证登录成功
```

**4. 等待操作**
```
操作类型：等待
等待时间：2000 毫秒
步骤描述：等待页面加载
```

#### 4.3.4 断言类型详解

| 断言类型 | 说明 | 期望值示例 |
|---------|------|-----------|
| 文本包含 | 验证元素文本包含指定内容 | "成功" |
| 文本等于 | 验证元素文本完全匹配 | "登录成功" |
| 元素可见 | 验证元素在页面上可见 | 无需期望值 |
| 元素存在 | 验证元素在 DOM 中存在 | 无需期望值 |
| 属性值 | 验证元素的属性值 | "active" |

### 4.4 动态变量使用

#### 4.4.1 内置变量

平台提供丰富的内置变量，支持测试数据动态化：

| 变量 | 说明 | 示例 |
|------|------|------|
| `{random_phone()}` | 生成随机手机号 | 13812345678 |
| `{random_email()}` | 生成随机邮箱 | test123@example.com |
| `{random_string(n)}` | 生成随机字符串（n 位） | abc123 |
| `{random_number(n)}` | 生成随机数字（n 位） | 123456 |
| `{timestamp()}` | 当前时间戳 | 1672531200 |
| `{date()}` | 当前日期 | 2024-01-01 |
| `{datetime()}` | 当前日期时间 | 2024-01-01 12:00:00 |

#### 4.4.2 使用数据工厂

1. 在输入值字段，点击 **引用数据工厂** 图标
2. 选择已创建的数据工厂记录
3. 系统会自动填充对应的数据

#### 4.4.3 变量使用示例

**场景**：注册新用户

```
步骤 1：输入文本
  目标元素：用户名输入框
  输入值：user_{random_string(6)}
  
步骤 2：输入文本
  目标元素：邮箱输入框
  输入值：{random_email()}
  
步骤 3：输入文本
  目标元素：手机号输入框
  输入值：{random_phone()}
```

### 4.5 步骤编辑技巧

#### 4.5.1 拖拽排序

1. 鼠标悬停在步骤上
2. 按住拖拽手柄（⋮⋮图标）
3. 拖动步骤到目标位置
4. 释放鼠标完成排序

#### 4.5.2 步骤展开/折叠

- 点击步骤右侧的 **展开/折叠** 按钮
- 支持批量展开/折叠所有步骤
- 折叠状态便于快速浏览，展开状态便于详细编辑

#### 4.5.3 步骤复制

1. 鼠标悬停在步骤上
2. 点击 **复制** 按钮
3. 在目标位置粘贴

#### 4.5.4 步骤删除

1. 点击步骤右侧的 **删除** 按钮
2. 确认删除操作

### 4.6 用例执行

#### 4.6.1 执行配置

在用例详情页，配置执行参数：

| 配置项 | 说明 | 选项 |
|-------|------|------|
| 测试引擎 | 选择自动化引擎 | Playwright（推荐）、Selenium |
| 浏览器 | 选择浏览器类型 | Chrome、Firefox、Safari、Edge |
| 运行模式 | 选择执行模式 | 有头模式、无头模式 |

**执行模式说明**：
- **有头模式**：显示浏览器窗口，可实时查看执行过程，适合调试
- **无头模式**：后台执行，不显示浏览器窗口，执行速度更快，适合批量执行

#### 4.6.2 执行测试

1. 确认执行配置
2. 点击 **运行** 按钮
3. 系统开始执行测试，实时显示执行日志
4. 执行完成后显示结果

#### 4.6.3 查看执行结果

执行完成后，可查看：

1. **执行状态**：通过/失败
2. **执行时长**：总耗时（秒）
3. **执行日志**：详细的步骤执行记录
4. **截图证据**：关键步骤的页面截图
5. **错误信息**：失败步骤的错误详情

#### 4.6.4 重新运行

1. 如果用例执行失败，可点击 **重新运行**
2. 系统会使用相同的配置重新执行
3. 适合用于验证问题修复

### 4.7 用例管理

#### 4.7.1 用例搜索

支持多维度搜索：

- **关键字搜索**：搜索用例名称、描述
- **项目筛选**：按项目过滤
- **优先级筛选**：按优先级过滤
- **状态筛选**：草稿/就绪/执行中/通过/失败

#### 4.7.2 用例复制

1. 在用例列表，点击 **复制** 按钮
2. 系统会创建用例的副本（包含所有步骤）
3. 修改副本信息，保存为新用例

#### 4.7.3 用例删除

1. 点击用例右侧的 **删除** 按钮
2. 确认删除操作
3. **注意**：已加入测试套件的用例删除前需谨慎

#### 4.7.4 用例状态管理

| 状态 | 说明 | 使用场景 |
|------|------|---------|
| 草稿 | 正在编辑，尚未完成 | 用例设计阶段 |
| 就绪 | 已完成，可执行 | 可加入测试套件 |
| 执行中 | 正在执行 | 系统自动管理 |
| 通过 | 最近执行通过 | 测试稳定 |
| 失败 | 最近执行失败 | 需检查修复 |

## 第五章 测试套件管理

### 5.1 测试套件概述

测试套件是测试用例的集合，用于批量执行测试。通过测试套件，可以：
- 组织相关测试用例
- 自定义执行顺序
- 批量执行测试
- 统一配置执行环境

### 5.2 创建测试套件

#### 5.2.1 基本信息

1. 进入 **套件管理** 页面
2. 点击 **新增套件**
3. 填写套件信息：

| 字段 | 说明 | 示例 |
|------|------|------|
| 套件名称 | 简洁描述套件用途 | 登录模块测试套件 |
| 套件描述 | 详细描述套件范围 | 包含登录相关的所有测试用例 |
| 所属项目 | 选择套件所属项目 | 电商平台测试 |

4. 点击 **保存** 进入用例选择

#### 5.2.2 选择测试用例

1. **可用用例列表**：显示当前项目下所有"就绪"状态的用例
2. **搜索用例**：通过关键字快速查找用例
3. **添加用例**：
   - 点击用例右侧的 **添加** 按钮（→）
   - 或直接双击用例行
4. **已选用例列表**：显示已添加到套件的用例

#### 5.2.3 调整执行顺序

1. 在已选用例列表中，可调整用例顺序
2. **上移/下移**：点击用例右侧的上/下箭头
3. **删除**：点击删除按钮（×）移除用例
4. 执行顺序从上到下，序号小的先执行

#### 5.2.4 保存套件

1. 确认用例选择和顺序
2. 点击 **保存** 按钮
3. 系统显示套件详情

### 5.3 执行测试套件

#### 5.3.1 执行配置

点击套件的 **运行** 按钮，配置执行参数：

| 配置项 | 说明 | 选项 | 推荐 |
|-------|------|------|------|
| 测试引擎 | 自动化执行引擎 | Playwright、Selenium | Playwright |
| 浏览器 | 浏览器类型 | Chrome、Firefox、Safari、Edge | Chrome |
| 执行模式 | 显示模式 | 有头模式、无头模式 | 调试用有头，批处理用无头 |

**引擎选择建议**：
- **Playwright**：性能更好，支持更多特性，推荐使用
- **Selenium**：兼容性好，适合特定场景

**执行模式建议**：
- **有头模式**：调试测试、查看执行过程
- **无头模式**：定时任务、CI/CD 集成、批量执行

#### 5.3.2 开始执行

1. 确认执行配置
2. 点击 **开始执行** 按钮
3. 系统按顺序执行套件中的用例
4. 实时显示执行进度

#### 5.3.3 执行过程监控

执行过程中可查看：

1. **执行进度**：当前执行到第几个用例
2. **实时日志**：每个步骤的执行日志
3. **统计信息**：已通过、已失败、剩余用例数
4. **执行截图**：关键步骤的页面截图

#### 5.3.4 执行完成

执行完成后显示：

1. **执行状态**：
   - ✅ 通过：所有用例通过
   - ❌ 失败：存在用例失败
2. **统计信息**：
   - 总用例数
   - 通过数（绿色显示）
   - 失败数（红色显示）
3. **执行时长**：总耗时（秒）
4. **通过率**：通过用例占比

### 5.4 套件管理

#### 5.4.1 编辑套件

1. 在套件列表，点击 **编辑** 按钮
2. 修改套件信息：
   - 基本信息（名称、描述）
   - 用例组成（添加/删除/调整顺序）
3. 点击 **保存** 提交修改

#### 5.4.2 套件复制

1. 点击套件右侧的 **复制** 按钮
2. 系统创建套件副本（包含用例关联）
3. 修改副本信息，保存为新套件

#### 5.4.3 套件删除

1. 点击套件右侧的 **删除** 按钮
2. 确认删除操作
3. **注意**：删除套件不会影响已生成的执行记录

#### 5.4.4 套件搜索

支持多维度搜索：

- **关键字搜索**：搜索套件名称、描述
- **项目筛选**：按项目过滤
- **执行状态筛选**：未执行/通过/失败/执行中

### 5.5 套件执行统计

#### 5.5.1 查看统计信息

在套件列表中显示：

- **包含用例数**：套件中的用例总数
- **执行状态**：最近一次执行的状态
- **通过数**：最近执行通过的用例数
- **失败数**：最近执行失败的用例数

#### 5.5.2 执行历史

1. 点击套件名称进入详情页
2. 查看 **执行历史** 标签页
3. 显示所有执行记录：
   - 执行时间
   - 执行状态
   - 执行时长
   - 通过率

#### 5.5.3 趋势分析

通过执行历史，可分析：

- **质量趋势**：通过率是否稳定
- **稳定性**：是否频繁失败
- **执行效率**：执行时长变化

## 第六章 定时任务管理

### 6.1 定时任务概述

定时任务功能支持自动执行测试套件或用例，无需手动触发。适用于：
- 每日构建测试
- 定期回归测试
- 夜间批量执行
- CI/CD 流程集成

### 6.2 创建定时任务

#### 6.2.1 基本信息

1. 进入 **定时任务** 页面
2. 点击 **新增任务**
3. 填写任务信息：

| 字段 | 说明 | 示例 |
|------|------|------|
| 任务名称 | 简洁描述任务 | 每日登录测试 |
| 任务描述 | 详细描述任务目的 | 每天上午 9 点执行登录模块测试 |
| 关联项目 | 选择任务所属项目 | 电商平台测试 |

#### 6.2.2 任务类型

选择任务类型：

| 类型 | 说明 | 配置项 |
|------|------|-------|
| **测试套件执行** | 执行整个测试套件 | 选择测试套件 |
| **测试用例执行** | 执行指定的测试用例 | 选择一个或多个用例 |

**选择测试套件**：
- 从下拉列表选择已创建的测试套件
- 执行时会按套件中的顺序执行所有用例

**选择测试用例**：
- 支持多选
- 可自由选择不同套件中的用例
- 适合跨套件的专项测试

#### 6.2.3 触发器类型

平台支持三种触发方式：

**1. Cron 表达式**

最灵活的定时方式，支持精确到分钟的计划：

```
格式：分 时 日 月 周

示例：
0 9 * * *        # 每天上午 9 点执行
0 */2 * * *      # 每 2 小时执行一次
0 9 * * MON      # 每周一上午 9 点执行
0 9 1 * *        # 每月 1 号上午 9 点执行
```

**常用 Cron 表达式**：

| 表达式 | 说明 |
|-------|------|
| `0 * * * *` | 每小时执行 |
| `0 9 * * *` | 每天上午 9 点 |
| `0 9 * * MON-FRI` | 工作日每天上午 9 点 |
| `0 9 1,15 * *` | 每月 1 号和 15 号上午 9 点 |
| `0 */4 * * *` | 每 4 小时执行 |

**2. 固定间隔**

按固定时间间隔循环执行：

- **间隔时间**：最小 60 秒
- **执行逻辑**：从任务创建时间开始，每隔指定时间执行一次
- **适用场景**：高频次测试、监控测试

**3. 单次执行**

在指定时间执行一次：

- **执行时间**：选择具体的日期时间
- **适用场景**：临时测试、计划性测试

#### 6.2.4 执行配置

配置测试执行环境：

| 配置项 | 说明 | 选项 |
|-------|------|------|
| 执行引擎 | 自动化引擎 | Playwright、Selenium |
| 浏览器类型 | 浏览器 | Chrome、Firefox、Edge |
| 无头模式 | 是否后台执行 | 是/否 |

#### 6.2.5 通知配置

配置测试完成后的通知：

**1. 通知时机**

- ✅ 成功时通知：测试通过时发送通知
- ✅ 失败时通知：测试失败时发送通知

**2. 通知方式**

| 方式 | 说明 | 配置项 |
|------|------|-------|
| **邮箱通知** | 发送邮件 | 选择收件人 |
| **Webhook 机器人** | 推送消息到 IM | 需预先配置 Webhook |
| **两者都发送** | 同时使用两种方式 | 配置邮箱和 Webhook |

**3. 选择收件人**

- 从用户列表中选择
- 支持多选
- 建议选择项目成员和相关干系人

#### 6.2.6 保存任务

1. 确认所有配置信息
2. 点击 **保存** 按钮
3. 任务创建成功，状态为"激活"

### 6.3 任务管理

#### 6.3.1 查看任务列表

任务列表显示：

- **任务名称**：任务的名称
- **任务类型**：套件执行/用例执行
- **触发类型**：Cron/间隔/单次
- **状态**：激活/暂停/已完成/失败
- **执行引擎**：Playwright/Selenium
- **浏览器**：Chrome/Firefox/Edge
- **下次运行时间**：计划执行时间
- **上次运行时间**：最近一次执行时间

#### 6.3.2 任务筛选

支持多维度筛选：

- **任务类型**：测试套件/测试用例
- **触发类型**：Cron/间隔/单次
- **状态**：激活/暂停/已完成/失败

#### 6.3.3 编辑任务

1. 点击任务的 **更多** > **编辑**
2. 修改任务配置
3. 点击 **保存** 提交修改

**注意**：修改触发器配置后，下次执行时间会重新计算。

#### 6.3.4 暂停/恢复任务

**暂停任务**：
1. 点击任务的 **更多** > **暂停**
2. 任务状态变为"暂停"
3. 暂停期间不会自动执行

**恢复任务**：
1. 点击任务的 **更多** > **恢复**
2. 任务状态变为"激活"
3. 恢复后按配置继续执行

#### 6.3.5 立即执行

临时触发任务执行：

1. 点击任务的 **立即执行** 按钮
2. 系统立即执行一次任务
3. 不影响原定的定时计划

#### 6.3.6 删除任务

1. 点击任务的 **更多** > **删除**
2. 确认删除操作
3. **注意**：删除后无法恢复，历史执行记录保留

### 6.4 任务执行

#### 6.4.1 自动执行

系统会自动执行激活状态的定时任务：

1. **调度器**：后台服务每分钟检查一次
2. **执行条件**：当前时间 >= 下次运行时间
3. **执行流程**：
   - 检查任务状态（必须为"激活"）
   - 执行测试（套件或用例）
   - 记录执行结果
   - 发送通知（如配置）
   - 更新下次运行时间

#### 6.4.2 执行记录

每次执行都会生成执行记录：

- **执行时间**：开始和结束时间
- **执行状态**：通过/失败
- **执行结果**：详细的步骤执行日志
- **统计信息**：通过数、失败数

#### 6.4.3 通知发送

执行完成后，根据配置发送通知：

**邮件通知内容**：
- 任务名称
- 执行时间
- 执行状态
- 统计信息（总用例数、通过数、失败数）
- 执行报告链接

**Webhook 通知**：
- 推送 JSON 格式的执行结果
- 支持飞书、企微、钉钉等机器人

### 6.5 最佳实践

#### 6.5.1 定时策略

**每日构建**：
```
Cron: 0 9 * * *
说明：每天上午 9 点执行
场景：日常回归测试
```

**高频监控**：
```
间隔：3600 秒（1 小时）
说明：每小时执行一次
场景：核心功能监控
```

**工作日测试**：
```
Cron: 0 9 * * MON-FRI
说明：工作日每天上午 9 点执行
场景：工作日定时测试
```

#### 6.5.2 通知策略

**关键任务**：
- 成功和失败都通知
- 通知所有项目成员
- 使用邮件 + Webhook 双重通知

**常规任务**：
- 仅失败时通知
- 通知负责人
- 使用邮件通知

#### 6.5.3 任务命名规范

建议采用统一的命名规范：

```
[项目]-[模块]-[频率]-[类型]

示例：
电商 - 登录 - 每日 - 回归
电商 - 支付 - 每小时 - 监控
```

## 第七章 测试报告

### 7.1 报告概述

每次测试执行完成后，系统会自动生成测试报告，包含：
- 执行概览
- 详细结果
- 执行日志
- 截图证据

### 7.2 查看执行记录

#### 7.2.1 执行记录列表

进入 **执行记录** 页面，查看历史执行：

| 字段 | 说明 |
|------|------|
| 执行 ID | 执行记录编号 |
| 测试套件/脚本 | 执行的对象 |
| 执行环境 | 浏览器类型 |
| 状态 | 执行结果（成功/失败） |
| 总用例数 | 执行的用例总数 |
| 通过数 | 通过的用例数 |
| 失败数 | 失败的用例数 |
| 执行时长 | 总耗时（秒） |
| 执行时间 | 开始执行的时间 |

#### 7.2.2 筛选执行记录

支持多维度筛选：

- **项目**：按项目过滤
- **状态**：成功/失败/执行中
- **时间范围**：开始和结束日期
- **测试套件**：按套件筛选

### 7.3 查看报告详情

点击执行记录的 **查看** 按钮，进入报告详情页。

#### 7.3.1 执行概览

显示整体执行情况：

- **执行状态**：✅ 成功 / ❌ 失败
- **执行时长**：总耗时（秒）
- **用例统计**：
  - 总用例数
  - 通过数（绿色）
  - 失败数（红色）
  - 跳过数（灰色）
- **通过率**：通过用例占比（百分比）

#### 7.3.2 用例详情

逐个显示每个用例的执行结果：

**1. 用例基本信息**
- 用例名称
- 用例描述
- 优先级
- 执行状态

**2. 步骤执行详情**
- 步骤序号
- 操作类型
- 执行状态
- 执行时长
- 错误信息（如失败）

**3. 执行日志**
- 详细的操作步骤日志
- 时间戳
- 日志级别（INFO/WARNING/ERROR）

**4. 截图证据**
- 关键步骤的页面截图
- 失败步骤的错误截图
- 支持放大查看

#### 7.3.3 错误分析

对于失败的用例，提供详细的错误信息：

- **错误类型**：如 TimeoutError、AssertionError 等
- **错误消息**：具体的错误描述
- **错误位置**：失败的步骤
- **堆栈跟踪**：完整的错误堆栈

### 7.4 报告导出

#### 7.4.1 导出格式

支持导出为多种格式：

- **PDF**：适合存档和分享
- **HTML**：适合在线查看
- **Excel**：适合数据分析

#### 7.4.2 导出步骤

1. 在报告详情页，点击 **导出** 按钮
2. 选择导出格式
3. 系统生成文件并下载

### 7.5 报告分析

#### 7.5.1 趋势分析

通过查看历史报告，分析：

- **质量趋势**：通过率是否提升
- **稳定性**：失败是否集中在特定模块
- **执行效率**：执行时长是否优化

#### 7.5.2 问题定位

利用报告快速定位问题：

1. **查看失败步骤**：定位具体失败的步骤
2. **分析错误日志**：理解失败原因
3. **查看截图**：直观了解页面状态
4. **复现问题**：单独执行失败用例

#### 7.5.3 报告分享

- **分享链接**：生成报告链接分享给团队成员
- **邮件发送**：通过邮件发送报告
- **导出存档**：导出 PDF 存档备查

## 第八章 最佳实践

### 8.1 元素管理最佳实践

#### 8.1.1 定位策略选择优先级

1. **ID 定位**（最优）
   ```css
   #username
   ```
   - 唯一性好
   - 执行速度快
   - 稳定性高

2. **CSS Selector**（推荐）
   ```css
   .btn-primary
   input[type="submit"]
   ```
   - 语法简洁
   - 性能较好
   - 适用范围广

3. **XPath**（备选）
   ```xpath
   //button[text()='登录']
   ```
   - 功能强大
   - 支持文本匹配
   - 性能相对较慢

4. **其他定位**（慎用）
   - Name、Class 等可能不唯一
   - 需配合其他条件使用

#### 8.1.2 元素命名规范

**推荐命名格式**：
```
[页面]_[元素类型]_[功能]_[状态]

示例：
login_btn_submit        # 登录页_按钮_提交
login_input_username    # 登录页_输入框_用户名
home_nav_menu_products  # 首页_导航_菜单_产品
```

**命名原则**：
- 简洁明了
- 见名知义
- 统一风格
- 避免重复

#### 8.1.3 元素分组策略

**按模块分组**：
```
登录模块
  - 登录页元素
  - 注册页元素
  - 找回密码页元素

商品模块
  - 商品列表页元素
  - 商品详情页元素
  - 购物车页元素
```

**按页面分组**：
```
首页元素
  - 头部元素
  - 导航元素
  - 底部元素

个人中心页元素
```

#### 8.1.4 备用定位器配置

为关键元素配置备用定位器：

```json
[
  {
    "strategy": "css",
    "value": "#submit-btn"
  },
  {
    "strategy": "xpath",
    "value": "//button[@type='submit']"
  },
  {
    "strategy": "css",
    "value": "button.btn-primary"
  }
]
```

**优势**：
- 提高元素定位成功率
- 降低页面变化影响
- 增强测试稳定性

### 8.2 测试用例设计最佳实践

#### 8.2.1 用例设计原则

**1. 单一职责**
- 每个用例只测试一个功能点
- 避免用例过长过复杂
- 便于问题定位

**2. 独立性**
- 用例之间无依赖
- 可单独执行
- 执行顺序不影响结果

**3. 可重复性**
- 多次执行结果一致
- 不依赖特定数据
- 使用动态变量

**4. 完整性**
- 覆盖正常场景
- 覆盖异常场景
- 覆盖边界条件

#### 8.2.2 用例结构建议

**推荐的用例步骤结构**：

```
1. 前置准备
   - 打开浏览器
   - 访问系统
   
2. 测试操作
   - 执行核心操作
   - 输入测试数据
   
3. 结果验证
   - 断言预期结果
   - 截图保存证据
   
4. 清理工作
   - 退出登录
   - 关闭浏览器
```

#### 8.2.3 断言设计

**断言要点**：
- 每个关键步骤后添加断言
- 断言预期结果明确
- 使用多种断言类型

**示例**：
```
步骤 5：断言
  断言类型：文本包含
  期望值：登录成功
  说明：验证登录操作成功

步骤 6：断言
  断言类型：元素可见
  期望值：用户头像
  说明：验证已跳转到首页
```

#### 8.2.4 数据驱动

**使用动态变量**：
```
用户名：user_{random_string(6)}
邮箱：{random_email()}
手机号：{random_phone()}
验证码：{random_number(6)}
```

**优势**：
- 避免数据冲突
- 支持并发执行
- 提高测试覆盖率

### 8.3 测试套件设计最佳实践

#### 8.3.1 套件组织策略

**按模块组织**：
```
登录模块测试套件
  - 正常登录测试
  - 错误密码测试
  - 忘记密码测试

商品模块测试套件
  - 商品搜索测试
  - 商品详情测试
  - 加入购物车测试
```

**按优先级组织**：
```
P0 核心用例套件
  - 登录
  - 下单
  - 支付

P1 重要用例套件
  - 搜索
  - 收藏
  - 评价
```

**按执行频率组织**：
```
每日构建套件
  - 核心功能测试

每周回归套件
  - 全量功能测试
```

#### 8.3.2 执行顺序优化

**排序原则**：
1. 前置依赖用例优先
2. 高优先级用例优先
3. 执行快的用例优先
4. 不稳定的用例靠后

**示例顺序**：
```
1. 登录测试（前置依赖）
2. 搜索商品（独立）
3. 商品详情（独立）
4. 加入购物车（依赖登录）
5. 提交订单（依赖购物车）
6. 支付测试（依赖订单）
```

### 8.4 定时任务最佳实践

#### 8.4.1 定时策略

**核心功能监控**：
```
触发类型：固定间隔
间隔时间：3600 秒（1 小时）
执行用例：P0 核心用例
通知方式：失败时 Webhook 通知
```

**每日回归测试**：
```
触发类型：Cron 表达式
Cron: 0 9 * * *（每天上午 9 点）
执行套件：每日构建套件
通知方式：失败时邮件通知
```

**全量回归测试**：
```
触发类型：Cron 表达式
Cron: 0 2 * * SUN（每周日凌晨 2 点）
执行套件：全量回归套件
通知方式：成功和失败都邮件通知
```

#### 8.4.2 通知策略

**分级通知**：

| 任务级别 | 通知时机 | 通知方式 | 通知对象 |
|---------|---------|---------|---------|
| P0 核心 | 失败时 | Webhook+ 邮件 | 全体项目成员 |
| P1 重要 | 失败时 | 邮件 | 项目负责人 |
| P2 常规 | 失败时 | 邮件 | 测试负责人 |
| P3 一般 | 不通知 | - | - |

### 8.5 执行优化建议

#### 8.5.1 提升执行速度

**1. 使用无头模式**
- 无头模式比有头模式快 20-30%
- 适合批量执行和定时任务

**2. 优化等待时间**
- 使用智能等待（等待元素）
- 避免固定长等待
- 合理设置超时时间

**3. 并发执行**
- 将用例分散到多个套件
- 并行执行不同套件
- 提高总体执行效率

**4. 优化断言**
- 减少不必要的断言
- 使用高效的断言类型
- 避免复杂的断言逻辑

#### 8.5.2 提高稳定性

**1. 元素定位优化**
- 使用唯一性高的定位器
- 配置备用定位器
- 定期验证元素有效性

**2. 等待策略优化**
- 使用显式等待代替隐式等待
- 等待关键元素加载完成
- 设置合理的超时时间

**3. 异常处理**
- 添加重试机制
- 捕获并记录异常
- 提供清晰的错误信息

**4. 环境隔离**
- 使用独立测试环境
- 避免多任务并发冲突
- 数据隔离和清理

### 8.7 常见问题处理

#### 8.7.1 元素定位失败

**问题表现**：
- 执行时报"找不到元素"错误
- 验证元素时提示无效

**解决方案**：
1. 检查定位表达式是否正确
2. 使用浏览器开发者工具验证
3. 添加备用定位器
4. 增加等待时间
5. 检查元素是否在 iframe 中

#### 8.7.2 测试执行超时

**问题表现**：
- 用例执行时间过长
- 报 TimeoutError 错误

**解决方案**：
1. 优化等待策略，使用智能等待
2. 减少固定等待时间
3. 检查是否有死循环
4. 增加超时时间配置

#### 8.7.3 测试结果不稳定

**问题表现**：
- 同一用例有时通过有时失败
- 失败原因不一致

**解决方案**：
1. 检查是否有数据依赖
2. 使用动态变量生成测试数据
3. 添加充分的前置条件检查
4. 优化断言逻辑
5. 添加重试机制

#### 8.7.4 页面加载慢

**问题表现**：
- 页面元素加载不及时
- 操作失败率高

**解决方案**：
1. 使用"等待元素"操作
2. 增加页面加载等待
3. 优化网络环境
4. 使用性能更好的浏览器
